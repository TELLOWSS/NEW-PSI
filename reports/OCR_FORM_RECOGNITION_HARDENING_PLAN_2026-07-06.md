# NEW-PSI OCR 양식 인식 정확도 개선 검토 및 실행계획

작성일: 2026-07-06

## 1. 결론

현재 문제는 OCR 엔진 하나의 성능 문제가 아니라, `고정 양식 판독값`과 `문항 답변 의미해석`을 한 번에 맡긴 구조에서 발생한다.

특히 현재 NEW-PSI 양식에서는 하단에 `공종`, `현장 등록 한글이름`, `서명` 위치가 고정되어 있고, Q1은 “이번 작업에서 가장 위험한 작업”을 적는 칸이다. Q1 답변은 근로자가 실제로 하는 작업 중 가장 위험한 세부작업 또는 위험작업이며, 위험분석·점수·TBM 환류의 핵심 근거다.

다만 데이터 구조에서는 `공종`과 `Q1 실제 위험작업`을 분리해야 한다. 예를 들어 `공종=형틀`, `Q1=거푸집 해체 중 단부 추락 위험`처럼 상위 분류와 실제 위험작업을 함께 보존해야 한다. Q1을 버리거나 약하게 보는 것이 아니라, Q1을 공종 칸의 대체값으로 덮어쓰지 않는 것이 핵심이다.

따라서 상품화 수준으로 가려면 OCR 엔진만 믿는 방식이 아니라 다음 3단계 구조가 필요하다.

1. 고정칸 위치 판독: 공종/이름/서명/날짜는 지정 영역에서만 읽는다.
2. 문항 답변 분리: Q1은 실제 위험작업, Q2는 위험요인, Q3는 위험수준, Q4는 감소대책, Q5는 실천행동으로 각각 독립 저장한다.
3. 관리자 검증 후 확정: 신뢰도 낮은 공종/이름은 추정 확정하지 않고 검토 대기 처리한다.

## 2. 확인한 원인

### 양식 구조

- 공종과 이름은 하단 고정 위치에 있다.
- Q1은 근로자가 실제로 하는 작업 중 “가장 위험한 세부작업”을 적는 칸이다.
- 예: 공종이 `형틀`이면 Q1에는 `고소작업`, `자재 양중`, `거푸집 해체`, `단부 작업` 등이 들어갈 수 있다.
- Q1은 실제 작업 분석의 핵심값이지만, 하단 공종 칸의 확정값을 대체하지는 않는다.

### 기존 로직 위험

기존 후처리 로직은 `jobField`가 `기타` 또는 `미분류`처럼 애매할 때 전체 본문과 Q1~Q5 답변에서 `형틀`, `철근`, `전기`, `타설` 같은 단어를 찾아 공종을 보정할 수 있었다.

이 방식은 기존 자유 양식에서는 도움이 될 수 있지만, 현재 고정 양식에서는 Q1 답변을 공종으로 오인할 수 있다.

## 3. 즉시 반영한 개선

### 프롬프트 보강

서버 OCR 재분석 경로와 기존 Gemini 분석 경로에 다음 규칙을 추가했다.

- `jobField`는 페이지 하단 왼쪽 `공종` 칸에 실제로 적힌 값만 사용한다.
- `name`은 페이지 하단 가운데 `현장 등록 한글이름` 칸에 실제로 적힌 값만 사용한다.
- Q1 답변은 세부작업/위험작업 설명이며, 위험분석·점수·교육 환류의 핵심 근거로 사용한다.
- 단, Q1 답변을 하단 `공종` 칸의 확정값처럼 `jobField`에 덮어쓰지는 않는다.
- 하단 공종/이름 칸이 비었거나 흐리면 추정하지 않고 `jobField=미분류`, `name=식별 대기`로 둔다.

### 후처리 안전장치

후처리 로직에서 Q1~Q5 답변 내용을 근거로 하단 `공종` 확정값을 자동 보정하지 않도록 막았다.

또한 AI가 실수로 `jobField`에 `철근 조립 작업 중 낙하 위험`처럼 Q1 답변 문장을 넣으면, 이를 공종으로 확정하지 않고 `미분류`로 격리한다. 해당 문장은 `handwrittenAnswers[Q1]`에서 실제 위험작업으로 보존되어 분석에 사용된다.

## 5. 다음 고도화 계획

### 1단계: 고정 양식 좌표 기반 필드 추출

목표: 공종/이름/서명을 문서 전체 추론이 아니라 정해진 위치에서만 읽는다.

필요 기능:

- A4 기준 좌표 템플릿 정의
- 공종 영역, 현장 등록 한글이름 영역, 서명 영역 crop
- Q1~Q5 답변 영역 crop
- 각 영역별 OCR 결과와 전체 OCR 결과 분리 저장

판정 기준:

- 공종은 하단 공종 영역 OCR 결과만 사용
- 이름은 하단 현장 등록 한글이름 영역 OCR 결과만 사용
- Q1은 실제 위험작업 후보로 사용하되, 하단 공종 칸의 자동 대체값으로 사용하지 않음

### 2단계: 양식 정렬 보정

목표: 사진 촬영이나 스캔 기울어짐에도 같은 좌표로 읽을 수 있게 한다.

필요 기능:

- 페이지 테두리 또는 Q1~Q5 박스 기준 자동 정렬
- 회전/기울기 보정
- 여백 잘림 감지
- 저해상도/흐림/그림자 감지

운영 기준:

- 정렬 실패 시 자동 분석 확정 금지
- `이미지 상태 확인 필요`로 관리자 검토 대기

### 3단계: 공종 사전과 Q1 세부작업 사전 분리

목표: `공종`과 `위험작업`을 데이터 구조상 완전히 분리한다.

예시:

- 공종: 형틀
- Q1 실제 위험작업: 거푸집 해체, 단부 작업, 자재 양중
- Q2 위험요인: 추락, 낙하, 끼임

분석 원칙:

- 공종은 근로자 식별과 월별 추적 기준
- Q1은 근로자가 실제로 수행하는 위험작업 분석 기준
- Q2~Q5는 안전역량 지표와 교육 환류 기준

### 4단계: OCR 검증 패키지 자동 생성

목표: 실제 현장 샘플 50~100건으로 정확도를 숫자로 증명한다.

필요 항목:

- 공종 정확도
- 이름 정확도
- Q1~Q5 답변 추출률
- 체크박스 인식률
- 관리자 수정률
- 분석 확정률
- 반려 사유 분포

상품화 기준:

- 공종/이름 자동 확정 정확도 90% 이상
- Q1~Q5 답변 추출률 95% 이상
- 관리자 수정 후 분석 가능률 98% 이상
- 오류 기록은 자동 확정하지 않고 검토 대기 처리

## 6. 운영 판단

유료 OCR/API를 쓰면 정확도와 처리량은 개선될 수 있다. 그러나 유료 엔진만으로 이 문제를 해결하는 것은 위험하다.

상품화에 필요한 핵심은 `비싼 엔진`이 아니라 `양식 위치 기반 판독`, `문항 의미 분리`, `검증 실패 격리`, `관리자 확정 이력`이다.

따라서 다음 개발 우선순위는 엔진 교체가 아니라 고정 양식 인식 파이프라인을 만드는 것이다.

1. 하단 공종/이름 영역 좌표 추출
2. Q1~Q5 영역 좌표 추출
3. 관리자 검증 UI에서 원본 영역과 추출값 나란히 표시
4. 50~100건 검증 패키지로 정확도 표 생성

이 구조가 잡히면 무료/유료 엔진 어느 쪽을 쓰더라도 품질이 안정되고, 바이어에게도 “엔진 성능에만 의존하지 않는 안전한 분석 체계”로 설명할 수 있다.
