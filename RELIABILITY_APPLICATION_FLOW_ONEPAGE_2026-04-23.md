# RELIABILITY APPLICATION FLOW ONEPAGE (2026-04-23)

## 0) 목적
이 문서는 **신뢰성 규칙이 프로그램에 실제로 어떻게 적용되는지**를 운영 흐름 기준으로 설명합니다.
핵심은 점수 계산 하나가 아니라, **입력 → 평가 → 품질 게이트 → 감사 로그 → 릴리즈 검증 → KPI 모니터링**의 전체 체계입니다.

---

## 1) 적용 흐름(운영 순서)

### Step 1. 입력 수집 및 구조화
- 입력: 자유기술 응답, 작업 맥락, 위험요소, 조치 내용
- 필수 구조화 필드: `scoreBreakdown`, `actionHistory`, `auditTrail`
- 목적: 이후 모든 점수에 대해 “왜 그렇게 판정했는지”를 역추적 가능하게 만드는 기반 확보

### Step 2. 6지표 앵커 평가 실행
- 평가 방식: 키워드 매칭이 아닌 **A/B/C 앵커 기준 자유기술 평가**
- 지표: 심리지표, 업무이해도, 위험성평가, 숙련도, 개선이행도, 반복위반패널티
- 결과: 지표별 근거와 점수 생성 + `scoreReasoning` 기록

### Step 3. 일관성 게이트(`strict8`) 적용
- 규칙: 유사 맥락 근로자 비교 시 점수 편차 `±8` 이내
- 명령: `npm run check:score-consistency:strict8`
- 해석:
  - PASS: 운영 가능
  - FAIL: 캘리브레이션 및 원인분석 후 재검증

### Step 4. 감사 추적 강제
- 필수 기록: `scoreReasoning` 100%, `auditTrail` 100%
- 필수 성격: append-only 변경 이력
- 목적: 사후 분쟁 시 점수 변경 사유/시점/주체를 재현 가능하게 보장

### Step 5. 릴리즈 검증 파이프라인
- 사전검증: `npm run verify:fast`
- 릴리즈검증: `npm run verify:release`
- 목적: 규칙 변경 전/후에 같은 입력이 같은 품질 조건을 만족하는지 재현성 확인

### Step 6. 운영 KPI 자동 모니터링
- 일일 수집: `npm run kpi:collect-daily`
- 주간/월간 집계: KPI 대시보드 및 월간 보고서 반영
- 핵심 지표 예시:
  - strict8 PASS율
  - 근거 기록률(`scoreReasoning`)
  - 감사로그율(`auditTrail`)
  - 평가자 합치도
  - 비용절감율

---

## 2) 역할별 신뢰성 체감 지점

### 근로자 관점
- 점수마다 근거가 남아 “왜 이 점수인지” 설명 가능
- 동일 조건에서 점수 급변 방지(strict8)

### 평가자 관점
- 앵커 기준으로 해석 편차 축소
- 합치도/편차를 수치로 확인하며 캘리브레이션 가능

### 경영자 관점
- 품질(일관성) + 비용(절감) + 재현성(검증통과)을 월간 단위로 동시에 추적
- 의사결정을 KPI 기준으로 표준화

## 4) 실패 시 즉시 조치 규칙
1. strict8 하락(<90%): 유사군 재분류 + 앵커 예시 보강 + 재검증
2. 근거 기록률 하락(<95%): 배포 차단, 입력 필드 강제
3. 감사로그 누락 발생: append-only 로그 강제 후 재실행
4. 비용절감 하락(<20%): OCR 예외선별 기준 재점검

## 5) 결론
프로그램의 신뢰성은 단일 알고리즘 정확도가 아니라,
**근거기록(Explainability) + 일관성게이트(Consistency) + 감사추적(Auditability) + 운영성과(KPI)**를
한 흐름으로 강제하는 운영 시스템으로 적용되어 있습니다.
