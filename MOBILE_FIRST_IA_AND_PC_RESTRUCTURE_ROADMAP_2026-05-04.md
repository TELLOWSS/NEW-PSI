# MOBILE FIRST IA AND PC RESTRUCTURE ROADMAP (2026-05-04)

- 목적: 첨부 목업의 직관적 모바일 UX와 현재 NEW-PSI의 실제 기능 자산을 결합해, 모바일 우선 재구성과 PC 후속 재편의 실행 기준을 정의한다.
- 전략 요약: **모바일은 행동 중심**, **PC는 운영 중심**으로 분리 설계한다.
- 기본 판단: 현재 프로그램은 기능 자산이 강하고, 첨부 목업은 진입 UX가 강하다. 따라서 `기능은 유지`, `사용경험은 재설계`가 최적이다.

---

## 1) 핵심 결론

### 1-1. 왜 모바일 먼저인가
1. 현장 사용자는 기능 탐색보다 즉시 행동이 중요하다.
2. 현재 구조는 정보량이 많아 모바일에서 인지 부하가 높다.
3. 첨부 목업은 `첫 진입 → OCR 시작 → AI 리스크 확인` 흐름이 매우 짧고 명확하다.
4. 모바일 UX 기준이 먼저 고정돼야 이후 PC 재구성도 우선순위가 정리된다.

### 1-2. 왜 PC는 후속인가
- PC는 단순 축소판이 아니라 별도 목적 화면이어야 한다.
- 관리자/운영자/보고/배치 작업은 넓은 화면에서 생산성 중심으로 재구성하는 편이 낫다.
- 모바일 정보구조를 먼저 고정한 뒤, PC는 그 위에 관리 기능을 확장하는 방식이 안전하다.

---

## 2) 현재 프로그램과 목업의 콜라보 원칙

### 2-1. 유지할 것 (엔진)
- OCR 분석 엔진
- AI 리스크 산정 로직
- 보고서/기록 데이터 구조
- 관리자 승인/검토 흐름
- 다국어/10개국 QA 기준

### 2-2. 재설계할 것 (경험)
- 첫 화면 정보구조
- 모바일 네비게이션 방식
- CTA 우선순위
- 카드 밀도와 텍스트 길이
- 단계형 액션 흐름
- 테마별 시각 표현 방식

### 2-3. 해석 원칙
- 현재 프로그램은 `기능 플랫폼`
- 첨부 목업은 `모바일 사용경험 표준안`
- 따라서 목표는 "현재 기능을 버리는 것"이 아니라 "모바일에서 누구나 바로 쓰게 만드는 것"이다.

## 3) 모바일 우선 정보구조(IA)

## 3-1. 모바일 메인 탭 구조
1. `홈`
2. `분석`
3. `리포트`
4. `근로자`
5. `더보기`

### 3-2. 홈 화면 역할
- 오늘 상태 3초 이해
- 전체 현황 / 위험군 / 안전 상태 확인
- 최근 분석 리포트 확인
- `시작하기` CTA로 OCR 진입

### 3-3. OCR 화면 역할
- 수기 기록지 분석 시작
- 분석 전 체크리스트 확인
- 파일 업로드/카메라/권한 흐름 최소화

### 3-4. AI 리스크 화면 역할
- 리스크 점수 즉시 이해
- 적/황/녹 요약 확인
- 경향/개별 점검/AI 인사이트로 분기

### 3-5. 더보기 역할
- 설정
- 관리자 기능
- 고급 리포트
- 로그/운영 메뉴
- PC급 상세 기능으로 연결되는 보조 진입점

---

## 4) 모바일 UX 설계 원칙

### 4-1. 구조 원칙
- 첫 화면에서 CTA는 1개 우선 노출
- 핵심 수치는 3개 이하로 제한
- 상세 기능은 접기/펼치기 또는 다음 화면으로 분리
- 하단 탭바 고정
- 각 화면별 고정 CTA 1개 유지

### 4-2. 사용성 원칙
- 한 손 조작 기준
- 터치 타겟 44x44 이상
- 숫자 우선, 설명은 짧게
- 색상만으로 의미 전달 금지
- 단계별 진행 흐름 유지

### 4-3. 직관성 원칙
- `어디서 뭘 해야 하는지`를 설명하지 않아도 보이게 설계
- 기능명을 나열하지 말고 행동을 보여주기
  - 예: `OCR 분석`보다 `분석 시작`
  - 예: `AI 리스크`보다 `위험 확인`

### 4-4. 테마 원칙
- 다크 / 라이트 / 혼합 모드는 같은 구조를 공유
- 테마는 감성 차이, 구조는 고정
- 경고/위험 카드만 강조 강도 차등 적용

---

## 5) PC 후속 재구성 원칙

### 5-1. PC의 역할 재정의
PC는 모바일 확대판이 아니라 운영 콘솔이어야 한다.

### 5-2. PC 우선 기능군
- 대량 OCR 처리
- 필터/검색/정렬
- 관리자 승인/반려/재분석
- 보고서 생성/배포
- 근로자 이력/비교 분석
- 정책/품질/운영 로그

### 5-3. PC 재편 방향
- 대시보드: 요약 + 드릴다운 중심
- OCR: 배치 처리 중심
- AI 리스크: 비교/우선순위/후속조치 중심
- 리포트: 출력/배포/이력 중심
- 설정: 운영 규칙/권한/연동 중심

### 5-4. PC 레이아웃 원칙
- 좌측 기능 네비 + 우측 작업 영역
- 표/패널/필터 고정
- 모바일에서 숨긴 상세 데이터 적극 노출
- 멀티패널 구조 허용

대상:
- Dashboard(홈)
- OcrAnalysis
- PredictiveAnalysis

목표:
- 첨부 목업 수준의 진입 UX 확보
- 하단 탭/고정 CTA/핵심 요약 구조 정착
- 다크/라이트/혼합 모드 동일 구조 유지

완료 기준:
- 320/360/375/390 뷰포트 사용성 통과
- 첫 행동 도달 시간 단축
- 핵심 화면 3개 직관성 확보

### Phase 2. 모바일 연결 흐름 확장
대상:
- 최근 리포트
- 분석 결과 상세
- 알림/상태
- 진행중 작업 이어보기

목표:
- 홈 → OCR → AI 리스크 → 리포트 흐름 자연 연결
- 사용자가 길을 잃지 않게 단계 전환 정리

### Phase 3. PC 기능 재구성
대상:
- Dashboard
- OCR 운영
- AI 리스크 운영
- Reports
- Settings/Admin

목표:
- 모바일과 다른 목적의 운영 콘솔 확립
- 대량 처리와 관리자 생산성 극대화

### Phase 4. 공통 디자인 시스템 정식화
대상:
- 간격/타이포/색상/상태칩/버튼/카드/게이지
- 탭바/헤더/패널/리스트

목표:
- 모바일/PC 공통 토큰화
- 향후 기능 추가 시 일관성 유지

## 8) 성공 기준

### 모바일 성공 기준
- 처음 보는 사용자도 10초 내 시작 동작 가능
- 3개 핵심 화면에서 행동 버튼 위치가 일관됨
- 위험/주의/안전 상태를 텍스트 없이도 대략 이해 가능
- 실측 QA에서 오터치/겹침/가독성 이슈 최소화

### PC 성공 기준
- 운영자가 한 화면에서 더 많은 판단과 처리를 할 수 있음
- 배치 처리/필터/보고서 작업 시간이 감소
- 모바일과 역할이 겹치지 않음

## 9) 주의사항
- PC 구조를 모바일에 축소 이식하면 실패 가능성 높음
- 모바일에 현재 모든 기능을 그대로 노출하면 직관성 붕괴
- 테마보다 먼저 정보구조를 고정해야 함
- 기능 이름보다 행동 흐름을 먼저 설계해야 함

## 10) 최종 제안
- 지금 방향은 맞다.
- 모바일은 첨부 목업을 기준으로 `행동 중심 재설계`

- 현재 프로그램은 엔진으로 유지하고, 사용자 경험만 단계적으로 재구성하는 것이 가장 현실적이고 성공 확률이 높다.

## 13) 현장 실무자 기준 정보량 축소 운영계획 (즉시 적용)

### 13-1. 핵심 운영원칙
1. 한 화면에서 한 가지 판단만 하게 만든다. (위험 판단 / 즉시 조치 / 보고 완료)
2. 상단 1스크린 안에 `위험등급 + 지금 할 일 3개 + 담당/기한`만 우선 노출한다.
3. 설명형 문장은 기본 숨김 처리하고, 체크리스트형 행동 문구만 기본 노출한다.
4. 상세 데이터(근거/이력/추세)는 접기 기본값으로 제공한다.

### 13-2. 정보 우선순위 분류 (P0/P1/P2)
- P0 (즉시행동): 위험등급, 우선조치 3개, 담당자, 기한, 보고/확인 버튼
- P1 (판단보조): 점수 근거 요약 1~2줄, 전일 대비 변화, 경향 아이콘
- P2 (기록/분석): 상세 로그, 긴 해설, 과거 이력 전체, 고급 통계

원칙: 모바일 기본 화면에는 P0만 완전 노출, P1은 요약 노출, P2는 접힘/별도 진입.

### 13-3. 화면별 기본 노출 규칙
- 홈
  - 기본 노출: 오늘 위험등급, 오늘 우선조치 3개, 시작 버튼 1개
  - 접힘 처리: 부연 설명, 과거 추이 상세, 장문 인사이트
- OCR 분석
  - 기본 노출: 업로드/촬영 CTA, 분석 시작 버튼, 진행상태
  - 접힘 처리: 엔진 설명, 기술 로그, 세부 파라미터
- AI 리스크
  - 기본 노출: 위험버킷(R/Y/G), 즉시조치, 담당/기한
  - 접힘 처리: 상세 채점근거, 확장 해설, 이력 전체

### 13-4. 문구 압축 기준 (실무자용)
- 금지: 기능명 중심 문장 (예: "AI 리스크 분석 기능으로 진입")
- 권장: 행동 중심 문장 (예: "위험 확인", "즉시 조치", "보고 완료")
- 길이 기준:
  - 카드 제목 16자 이내
  - 본문 요약 28자 이내
  - 버튼 라벨 8자 이내

## 14) 2주 실행 플랜 (현장 테스트 포함)

### Week 1 (구조 고정)
1. 화면 요소 전수 분류: P0/P1/P2 태깅
2. 상단 요약 카드 고정: 위험등급/우선조치/담당/기한
3. 장문 영역 접힘 기본값 적용
4. CTA 단일화: 화면당 우선 버튼 1개

### Week 2 (검증/보정)
1. 현장 사용자 5명 태스크 테스트
2. 첫 판단까지 소요시간 측정
3. 오터치/재탐색 구간 로그 확인
4. 상위 3개 혼동 포인트 문구/배치 수정

## 15) 검증 KPI (실무자 체감 기준)

- 첫 판단 완료 시간: 30초 이내
- 첫 화면 스크롤 깊이: 기존 대비 40% 감소
- 즉시조치 버튼 도달률: 90% 이상
- 잘못된 경로 진입률: 20% 이상 감소
- 사용자 피드백 "정보가 많아 헷갈림": 50% 이상 감소

---

## 16) 운영 책임 분담

- 제품/기획: P0/P1/P2 우선순위 확정, 삭제/접힘 정책 승인
- 디자인: 1스크린 우선 레이아웃, 카드 밀도/타이포 축소
- 개발: 접힘 기본값, CTA 우선순위, 측정 이벤트 삽입
- 현장 리더: 테스트 참여자 모집, 실제 업무 시나리오 검증

운영 원칙: 기능 추가보다 정보 감량을 우선한다.
