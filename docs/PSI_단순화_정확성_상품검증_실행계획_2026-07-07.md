# NEW-PSI 단순화·정확성·상품검증 실행계획

작성일: 2026-07-07  
목적: PSI가 현재 정체성에 맞게 발전하고 있는지 확인하고, 겉으로는 쉽고 간편하지만 안쪽은 정밀한 상품으로 보이도록 검증·개선 절차를 고정한다.

---

## 1. 현재 판단 결론

현재 프로그램은 PSI의 핵심 방향과 대체로 일치하게 발전하고 있다.

확인된 핵심 방향은 다음과 같다.

```text
수기 위험성평가 기록지
-> OCR 분석
-> AI 구조화
-> 관리자 검증
-> 모국어 안내/개인 안전역량 재계산
-> 월별 추적관리/교육 환류
```

현재 코드와 검증 결과를 보면 다음 축은 이미 프로그램 안에 들어와 있다.

| 방향성 | 현재 반영 상태 | 판단 |
| --- | --- | --- |
| OCR 분석 결과만 별도 확인 | OCR-only scope 검사 통과 | 반영됨 |
| 월별 분류와 대량 데이터 완화 | 월별 index, month-first mode, deferred search 검사 통과 | 반영됨 |
| 접힘 기능과 개인별 삭제 | collapsible sections, delete actions 검사 통과 | 반영됨 |
| 관리자 수정 후 모국어 안내 동기화 | manager review sync 검사 통과 | 반영됨 |
| 관리자 수정 후 개인 안전역량 재계산 | competency profile recalculation 검사 통과 | 반영됨 |
| Q1과 공종 분리 | Q1/job-field separation 검사 통과 | 반영됨 |
| API 한도와 변경 양식 게이트 | quota/form gate 검사 통과 | 반영됨 |
| 점수 일관성 | strict ±8점 게이트 통과 | 반영됨 |
| 보안 헤더 | 7/7 통과 | 반영됨 |
| 빌드 가능성 | production build 통과 | 반영됨 |

다만 전체 빠른 검증은 `check:page-tone`에서 멈췄다.  
이는 핵심 로직 실패가 아니라 일부 화면의 색상/톤 표준화가 아직 남아 있다는 뜻이다.

상품성 관점에서는 이 항목도 중요하다. 이유는 단순하다.

> 사용자는 내부 로직보다 먼저 화면의 정돈감과 쉬움을 보고 정확성을 믿기 시작한다.

## 2. 검증 실행 결과

2026-07-07 기준 확인한 자동 검증 결과다.

| 검증 항목 | 결과 | 의미 |
| --- | --- | --- |
| `check:ocr-contract` | PASS | OCR 분석 계약이 화면·서비스·타입에 유지됨 |
| `check:ocr-record-view` | PASS | OCR 분석 결과만 별도 보기, 월별 탭, 점진 렌더링 존재 |
| `check:ocr-scale-view` | PASS | 대량 데이터 월별 모드와 검색 지연 처리 존재 |
| `check:ocr-collapse-delete` | PASS | 접힘 기능과 개인/근로자 삭제 기능 존재 |
| `check:manager-review-sync` | PASS | 관리자 수정 후 모국어 안내 점검과 역량 재계산 동기화 |
| `check:competency-weights` | PASS | W1~W5는 가중 점수, W6은 감점 배율로 분리 |
| `check:settings-direct-controls` | PASS | 시스템 설정에서 직접 확인·체크·변경 가능한 조작 노출 |
| `check:mobile-field-mode` | PASS | 모바일 현장모드 증명 패널, 고정 액션, 추적 앵커 존재 |
| `check:ocr-operational-hardening` | PASS | 대량 처리 보호, 실패 점수 제외, 날짜 정규화, Q1/공종 분리 보호 |
| `check:ocr-quota-form-gate` | PASS | 무료/유료 OCR 한도 표시와 변경 양식 Q1~Q5 게이트 보호 |
| `check:ocr-verification-package` | PASS | 개인정보 보호형 OCR 검증 패키지 export 가능 |
| `check:security-headers` | PASS | Vercel 보안 헤더 7/7 통과 |
| `check:ui-language` | PASS | UI 언어 감사 high 0, medium 2 |
| `check:report-language` | PASS | 리포트 14개 언어 검사 통과 |
| `check:core-metrics` | PASS | 핵심 지표 소비처 5/5 통과 |
| `check:score-consistency:strict8` | PASS | 유사 맥락 점수 편차 ±8점 게이트 통과 |
| `check:types` | PASS | 타입 검사 통과 |
| `build` | PASS | 배포 빌드 성공 |
| `verify:fast` | FAIL | `check:page-tone`에서 중단, 일부 화면 톤 표준화 67건 남음 |

중요한 보정도 수행했다.

- 기존 `check:ocr-operational-hardening` 검증은 예전 기준인 `본문 위험요인 기준 공종 보정`을 요구했다.
- 현재 PSI 원칙은 `Q1은 실제 위험작업으로 보존하되, 하단 공종을 대체하지 않는다`이다.
- 따라서 검증 기준을 `문항 답변으로 보이는 공종값 격리`로 갱신했다.
- 이후 `check:ocr-operational-hardening`과 `tests/ocrRecordNormalization.test.ts` 모두 통과했다.

## 3. 상품 경험은 3단계로 보여줘야 한다

PSI의 실제 내부 로직은 복잡하다.  
하지만 사용자에게는 3단계로 보여야 한다.

```text
1. 찍는다
2. 확인한다
3. 교육으로 보낸다
```

### 3.1 사용자에게 보이는 화면

| 단계 | 사용자 문장 | 화면에서 보여줄 것 |
| --- | --- | --- |
| 1. 찍는다 | 기록지를 촬영하거나 PDF를 넣는다 | 큰 버튼 1개: `기록지 촬영/분석` |
| 2. 확인한다 | 분석 결과에서 이상한 부분만 확인한다 | `정상`, `확인 필요`, `재촬영 권장` 3분류 |
| 3. 교육으로 보낸다 | 근로자에게 맞는 안내와 교육자료를 만든다 | `모국어 안내`, `개인 리포트`, `월별 계도자료` |

사용자가 처음 봤을 때의 인상은 다음이어야 한다.

> 복잡한 안전관리 데이터를 다루는데, 내가 누를 버튼은 많지 않다.

### 3.2 안쪽에 숨어 있어야 하는 로직

겉은 단순하지만 내부는 다음처럼 정밀해야 한다.

| 숨겨진 로직 | 역할 |
| --- | --- |
| 파일 형식·용량 검사 | 분석 전 실패를 줄임 |
| 촬영 품질 판정 | 흔들림, 잘림, 해상도 문제를 조기에 분리 |
| 양식 버전 인식 | 기존 양식과 변경 양식을 같은 Q1~Q5 구조로 맞춤 |
| 고정 영역 분리 | 이름, 공종, 날짜를 Q1 답변과 혼동하지 않음 |
| Q1 실제 위험작업 보존 | 위험 분석의 핵심 근거로 사용 |
| Q1/공종 분리 | 공종 통계 오염 방지 |
| 실패 기록 격리 | API 한도·이미지 오류가 정상 통계를 오염하지 않음 |
| 관리자 검증 | AI 판단을 현장 사실로 확정 |
| 모국어 안내 동기화 | 관리자 수정 내용이 외국인 근로자 안내에도 반영 |
| 개인 안전역량 재계산 | 수정된 해석 기준으로 점수와 6대 지표 갱신 |
| 월별 추적관리 | 단건 분석이 아니라 변화 증거로 축적 |

## 4. 정확하다는 것을 보여주는 증명 구조

정확성은 말로 주장하면 약하다.  
다음 5개 증거를 화면과 자료에서 반복해서 보여줘야 한다.

### 증거 1. 원문과 분석 결과가 나란히 보인다

분석 결과 옆에 원본 이미지 또는 OCR 원문을 보여준다.  
관리자는 “AI가 왜 이렇게 판단했는지” 바로 대조할 수 있어야 한다.

합격 기준:

- 원문 이미지/텍스트 접근 버튼이 모든 분석 기록에 있다.
- Q1~Q5 추출 결과와 원문 위치가 관리자 확인 흐름에서 분리되지 않는다.

### 증거 2. 확인 필요 항목만 먼저 보인다

모든 데이터를 길게 보여주면 정확해 보이지 않고 복잡해 보인다.  
정상 기록은 접고, 확인 필요한 기록만 먼저 떠야 한다.

합격 기준:

- 기본 화면은 `확인 필요`, `재촬영 권장`, `보호 우선` 중심으로 정렬한다.
- 정상 기록은 월별/공종별로 접어서 볼 수 있다.

### 증거 3. 관리자 수정이 흔적과 결과를 남긴다

관리자가 내용을 고치면 단순히 문장만 바뀌면 안 된다.

합격 기준:

- 수정 전/후 값이 남는다.
- 모국어 안내 상태가 바뀐다.
- 개인 안전역량 세부지표가 재계산된다.
- 승인 가능/보강 필요 상태가 표시된다.

### 증거 4. 같은 맥락은 비슷한 점수로 나온다

점수는 설득의 중심이다.  
유사한 답변이 크게 흔들리면 상품 신뢰가 무너진다.

현재 확인:

- 유사 맥락 점수 편차 ±8점 게이트 통과
- 시드 13건, 비교쌍 8건, 국적 커버리지 PASS

운영 기준:

- 정기 검증에서 ±8점 게이트를 유지한다.
- 새 양식/새 언어/새 공종이 추가되면 시드 데이터를 추가한다.

### 증거 5. 월별로 변화가 보인다

PSI의 설득력은 단건 판정이 아니라 추적관리에서 나온다.

합격 기준:

- 같은 근로자의 월별 점수 변화가 보인다.
- 같은 공종의 반복 위험요인이 보인다.
- 교육 전후 개선 여부가 표로 남는다.
- 보호 우선 대상이 줄었는지 볼 수 있다.

## 5. 지금 필요한 개선 우선순위

### P0. 첫 화면 경험 단순화

목표:

> 처음 보는 사람이 10초 안에 “아, 이건 기록지를 찍고 확인해서 교육으로 보내는 프로그램이구나”라고 이해하게 만든다.

개선:

- 모바일 현장모드 첫 화면을 `촬영/분석`, `확인 필요`, `교육 연결` 3개 중심으로 재정렬
- PC 화면은 `오늘 확인할 것`, `월별 보기`, `증빙 내보내기` 중심으로 압축
- 소개 화면은 실제 프로그램 화면과 다른 목업 느낌을 줄이고, 실제 운영 화면의 축약판으로 맞춤

검증:

- 비개발자 3명에게 30초 설명 없이 화면을 보여준다.
- “무엇을 하는 프로그램인지”를 말하게 한다.
- 3명 중 2명 이상이 `기록지 분석`, `위험 확인`, `교육/추적`을 말하면 통과.

### P0. Q1 실제 위험작업 표시 강화

목표:

> Q1은 버려지는 값이 아니라 실제 위험작업 분석의 핵심임을 화면에서 분명히 보여준다.

개선:

- OCR 상세 결과에 `공종`과 `실제 위험작업(Q1)`을 나란히 표시
- Q1이 공종으로 오인된 경우 `공종 미분류`, `Q1 실제 위험작업 보존` 안내 표시
- 리포트에는 `공종`, `세부 위험작업`, `위험요인` 3단 구조로 표시

검증:

- 변경 양식 샘플 30장 이상에서 Q1과 공종이 서로 덮어쓰이지 않는지 확인
- Q1이 위험 분석 사유에 반영되는지 확인

### P1. UI 톤 표준화 마감

목표:

> 전체 빠른 검증이 한 번에 통과하도록 만들고, 화면의 정돈감을 높인다.

현재 상태:

- `check:page-tone` 실패
- 남은 hardcoded tone pair 67건
- 주요 위치: `Introduction.tsx`, `SafetyChecks.tsx`, `WorkerManagement.tsx`

개선:

- 남은 색상/테두리 직접 지정값을 브랜드 톤 토큰으로 이동
- 소개 화면은 “브랜드 목업”보다 “실제 운영 증명판” 느낌으로 정리
- 사용자에게 보이는 버튼과 카드 수를 줄이고 핵심 행동만 남김

검증:

- `npm.cmd run check:page-tone` 통과
- 이후 `npm.cmd run verify:fast` 전체 통과

### P1. 현장 검증 전후 데이터 환류

목표:

> 사용자가 실증한 기록을 전후 비교표로 만들고, 향후 상품 증빙자료로 계속 쌓는다.

개선:

- OCR 분석 전/후, 관리자 수정 전/후, 교육 전/후 데이터를 CSV로 저장
- `report:field-validation-feedback`로 Markdown/JSON 보고서 생성
- 월별로 같은 양식으로 누적

검증:

- 최소 30건의 전후 검증표 생성
- 수정 전 점수, 수정 후 점수, 모국어 안내 상태, 승인 준비 상태가 모두 채워짐

### P2. 발표용 원포인트 증명 모드

목표:

> 공모전, 바이어 미팅, 내부 보고에서 2분 안에 PSI의 가치를 증명한다.

구성:

1. 기록지 1장 촬영
2. Q1~Q5 자동 분리
3. 공종과 Q1 실제 위험작업 분리 표시
4. 보호 우선/확인 필요 표시
5. 관리자 문장 수정
6. 모국어 안내와 개인 안전역량 재계산 표시
7. 월별 추적관리 화면으로 연결

검증:

- 2분 안에 시연 가능
- 발표자가 “OCR이 좋습니다”가 아니라 “기록이 교육과 추적관리로 돌아갑니다”라고 설명할 수 있어야 한다.

## 6. 구매자와 심사위원에게 보여줄 단순 메시지

### 첫 문장

PSI는 근로자가 직접 쓴 위험성평가 기록지를 촬영하면, 위험 인식 수준을 분석하고, 관리자가 확인한 뒤, 개인별 교육과 월별 추적관리로 이어주는 현장 안전지능 프로그램입니다.

### 세 문장 설명

1. 종이 기록지를 사진으로 찍으면 Q1~Q5 답변과 이름·공종·날짜를 분리합니다.
2. AI가 위험 인식과 안전역량을 분석하지만, 관리자가 최종 확인하고 수정할 수 있습니다.
3. 수정된 내용은 모국어 안내, 개인 안전역량, 월별 교육자료로 다시 환류됩니다.

### 한 장 증명

| 전 | 후 |
| --- | --- |
| 기록지는 보관만 됨 | 기록지가 위험 인식 데이터가 됨 |
| 교육 이해도 확인 어려움 | Q1~Q5로 이해 수준 확인 |
| 외국인 전달 한계 | 모국어 안내로 환류 |
| 단건 확인 중심 | 월별 추적관리 가능 |
| 관리자 경험에 의존 | 수정 이력과 점수 근거가 남음 |

## 7. 최종 합격 기준

PSI를 상품화 막바지 기준으로 합격시키려면 아래 기준을 통과해야 한다.

| 영역 | 합격 기준 |
| --- | --- |
| 쉬움 | 처음 보는 사용자가 `찍기 -> 확인 -> 교육` 흐름을 30초 안에 이해 |
| 정확성 | Q1~Q5 추출률 95% 이상, 공종/Q1 혼동률 3% 이하 |
| 신뢰성 | 관리자 수정 전후 이력과 재계산 결과가 남음 |
| 일관성 | 유사 답변 점수 편차 ±8점 이내 |
| 현장성 | 모바일에서 촬영 후 확인 필요/교육 연결까지 이어짐 |
| 추적성 | 동일 근로자 월별 변화와 교육 전후 비교가 가능 |
| 발표성 | 2분 시연으로 기록지 1장이 교육 환류까지 가는 장면을 보여줌 |
| 배포성 | `verify:fast`와 `build`가 모두 통과 |

## 9. 최종 판단

현재 PSI는 방향성을 잃은 상태가 아니다.  
오히려 핵심 기능은 정체성에 맞게 상당히 많이 들어와 있다.

지금부터의 핵심은 기능을 더 늘리는 것이 아니다.

> 복잡한 내부 로직을 사용자가 느끼지 않게 만들고, 단순한 화면 안에서 정확성과 추적관리의 증거를 보여주는 것이다.

상품화 막바지의 목표는 다음 한 문장이다.

> 사용자는 쉽게 찍고 확인하지만, 뒤에서는 공종·Q1·위험요인·관리자 검증·모국어 안내·개인역량·월별 추적이 하나의 흐름으로 정확하게 이어진다.
