# PSI Human Risk Engine 전환 계획 (2026-05-16)

- 작성일: 2026-05-16
- 목적: PSI를 기능 중심 앱이 아니라 `건설현장 인간 위험인지 운영체제`로 재정의하고, 앞으로 90일 실행 기준을 고정한다.
- 기준: 기존 6대 지표 체계, 현장 수기자료, 외국인 근로자 응답, 위험성평가 기록, 행동 전조 데이터를 하나의 구조로 통합한다.

---

## 1) 한 줄 정의

PSI의 핵심은 `문서 자동화`가 아니라, 현장 작업자가 `왜 위험한 판단을 했는지`를 구조화하고 예측 가능한 데이터로 바꾸는 것이다.

즉 앞으로의 PSI는 아래 흐름을 중심으로 운영한다.

`현장 수기/응답/행동` → `위험 Ontology` → `인지·판단 태깅` → `작업자 위험 프로파일 벡터` → `사고 전조 패턴` → `설명 가능한 예측/개입`

---

## 2) 전략 판단

### 기존 접근
- 수기 데이터를 OCR/LLM으로 읽어 점수화
- 문서 자동 생성과 보고 효율 중심
- 결과 점수의 일관성 확보에 초점

### 전환 접근
- 수기 데이터를 인간 판단 데이터로 구조화
- `사고 이후 기록`보다 `사고 이전 전조`에 초점
- 점수보다 `위험인지 벡터`와 `행동 패턴` 축적에 초점

### 결론
기존 6대 지표 체계는 폐기 대상이 아니라, `Human Risk Engine`의 입력/설명 레이어로 재정의해야 한다.

## 3) PSI 6대 지표 재정의안

| 기존 지표 | 운영상 새 해석 | 핵심 질문 | 저장 형태 예시 |
| --- | --- | --- | --- |
| 심리지표 | 책임성·집중도·경계상태 | 작업자가 지금 위험을 진지하게 받아들이는가 | `attention_state`, `risk_salience`, `ownership_level` |
| 업무이해도 | 작업 맥락 이해 | 내가 지금 무엇을 왜 어떤 순서로 하는지 이해하는가 | `task_understanding`, `sequence_understanding`, `tool_material_clarity` |
| 위험성평가 이해도 | 위험인지·원인 연결 능력 | 어떤 위험이 왜 발생하는지 설명 가능한가 | `hazard_recognition`, `cause_effect_reasoning`, `control_awareness` |
| 숙련도 | 실행 가능 행동 역량 | 작업 전·중·후 조치를 실제로 수행할 수 있는가 | `procedural_skill`, `verification_discipline`, `response_capability` |
| 개선이행도 | 시정조치 실행력 | 문제 발견 후 실제로 누가 언제 어떻게 고칠 것인가 | `corrective_action_readiness`, `owner_clarity`, `followthrough_level` |
| 반복위반 패널티 | 위험 정상화 신호 | 위험 행동이 습관화·정당화되고 있는가 | `risk_normalization`, `rule_bypass_tendency`, `habitual_violation_signal` |

### 운영 원칙
1. 총점은 유지하되, 내부 저장은 우선 `벡터` 중심으로 전환한다.
2. 점수는 보고용, 벡터는 학습/탐지용으로 분리한다.
3. 동일 점수라도 다른 프로파일을 허용한다.
   - 예: 숙련도 높음 + 위험 과소평가 높음
   - 예: 업무이해도 낮음 + 규정준수 의지는 높음

## 4) PSI 4엔진 구조

## 4-1. Human Risk Engine
가장 우선순위가 높은 엔진이다.

### 수집 축
- 작업 이해도
- 위험인지 수준
- 작업순서 이해
- 대응 행동 능력
- 동료/주변 위험 인지
- 시간압박/피로/과신/무감각

### 산출
- 개인 위험 프로파일 벡터
- 팀 위험 취약 프로파일
- 교육 필요 유형 분류

---

## 4-2. Site Context Engine
사람만으로 설명되지 않는 위험 조건을 구조화한다.

### 기본 필드
- 공정 단계
- 작업 종류
- 층수/구역
- 날씨
- 주간/야간
- 작업 밀도
- 장비 배치
- 외국인 비율
- 신입 비율
- 협력사/팀 조합

### 산출
- 맥락별 위험 증폭 계수
- 동일 작업의 현장 조건별 편차 분석

---

## 4-3. Behavior Pattern Engine
사고 이전 행동 반복을 수집한다.

### 관찰 예시
- 안전고리 미체결 반복
- 작업속도 급증
- 반복 실수 증가
- 특정 시간대 집중도 저하
- 동일 팀의 유사 위반 누적

### 산출
- 전조 시그널 목록
- 반복 패턴 군집
- 위험 증가 조합 후보

---

## 4-4. Predictive Risk Engine
설명 가능한 예측과 개입 우선순위를 제공한다.

### 예측 항목
- 사고 확률 상승 구간
- 집중관리 대상자/대상팀
- 공정별 위험 급증 가능성
- 위험 전이 가능성

### 출력 원칙
- 단순 `위험 높음`이 아니라 이유를 설명한다.
- 예: `야간 + 신입 비율 상승 + 작업순서 이해 부족 + 안전고리 미체결 증가`로 추락위험 상승

---

## 5) 위험 Ontology v1 초안

초기에는 5대 위험부터 고정한다.

| 대분류 | 중분류 | 세부 위험 | 핵심 관찰 포인트 |
| --- | --- | --- | --- |
| 추락 | 안전고리 | 미체결, 체결 불량, 걸이점 부적합 | 체결 이유 이해, 이동 중 해제 습관 |
| 추락 | 개구부/슬라브 | 미판개, 임시덮개 불량, 가장자리 접근 | 공간위험 인지, 우회 행동 여부 |
| 낙하 | 양중 | 하부통제 미흡, 낙하반경 진입 | 통제구역 이해, 신호수/유도 확인 |
| 낙하 | 적재 | 자재 고정 불량, 상하동시작업 | 적재 안정성 이해, 상부 위험 인지 |
| 협착 | 장비 | 회전반경 진입, 협소공간 동시작업 | 거리감각, 장비 사각지대 인지 |
| 협착 | 자재/거푸집 | 손 끼임, 임시 고정 불량 | 손 위치 인지, 보조도구 사용 |
| 붕괴 | 가시설 | 지지 불량, 임의 해체 | 구조안정 이해, 순서 위반 여부 |
| 붕괴 | 굴착 | 토사 붕락, 버팀 미흡 | 지반상태 인지, 접근거리 판단 |
| 감전 | 활선/임시전기 | 피복 손상, 차단 미확인 | 차단 절차 이해, 테스트 여부 |
| 감전 | 장비/케이블 | 접지 미흡, 습윤 환경 사용 | 환경위험 인지, 사용 전 점검 |

### Ontology 운영 원칙
- 위험명만 저장하지 않는다.
- `대분류 > 중분류 > 세부위험 > 원인 > 전조행동 > 통제행동`까지 연결한다.
- 향후 특허/리포트/대시보드는 이 Ontology를 공통 기준으로 참조한다.

## 6) 인간 판단 태그 체계 v1

점수보다 먼저 태그 체계를 고정한다.

| 태그군 | 태그 예시 |
| --- | --- |
| 위험인지 | 위험 과소평가, 위험 미인지, 원인 오해, 결과만 인식 |
| 작업이해 | 작업 목적 미이해, 순서 혼동, 도구 용도 불명확, 공정 맥락 부족 |
| 심리상태 | 시간압박, 피로, 주의분산, 과신, 무감각 |
| 규정행동 | 절차 생략, 우회행동, PPE 형식착용, 확인 생략 |
| 사회/소통 | 동료 의존, 지시 오해, 언어장벽, 신호 미공유 |
| 개선실행 | 책임자 불명확, 조치시점 불명확, 확인방법 부재, 재발방지 미흡 |

### 수기 예시 태깅
원문: `잠깐이라 괜찮다고 생각했다`
- `위험 과소평가`
- `시간압박`
- `우회행동`
- `위험 정상화 신호`

원문: `작업 순서는 알지만 왜 고리를 걸어야 하는지는 잘 모르겠다`
- `작업순서 부분이해`
- `원인 이해 부족`
- `통제수단 목적 미이해`

## 7) 데이터 구조화 스키마 초안

| 엔티티 | 목적 | 핵심 필드 |
| --- | --- | --- |
| `worker_profile_vector` | 개인 위험 프로파일 저장 | `workerId`, `taskUnderstanding`, `hazardRecognition`, `sequenceUnderstanding`, `riskNormalization`, `responseCapability` |
| `site_context_snapshot` | 시점별 현장 맥락 저장 | `siteId`, `trade`, `shiftType`, `weather`, `floorZone`, `teamDensity`, `newWorkerRatio`, `foreignWorkerRatio` |
| `behavior_event_log` | 전조 행동 이벤트 축적 | `timestamp`, `workerId`, `teamId`, `eventType`, `severity`, `repeatCount`, `relatedOntologyNode` |
| `judgment_tag_record` | 수기/응답의 판단 태깅 저장 | `sourceText`, `tags`, `confidence`, `reviewer`, `linkedMetric`, `linkedRiskType` |
| `risk_signal` | 예측 직전 시그널 집약 | `signalType`, `contributors`, `signalScore`, `explanation`, `recommendedAction` |

### 저장 원칙
- 원문은 보존하되, 운영의 중심은 태그/벡터/맥락 필드로 둔다.
- 한 건의 응답은 여러 태그를 동시에 가질 수 있다.
- 초기에는 규칙 기반 태깅 + 관리자 검토 혼합 운영이 적절하다.

## 11) 최종 판단

PSI의 경쟁력은 모델 크기나 UI 화려함이 아니라, `현장 인간 위험인지 데이터`를 얼마나 정교하게 구조화하느냐에 달려 있다.

따라서 앞으로의 개발 기준은 아래로 고정한다.

> 기능을 더 만드는 것보다, 위험 판단이 무너지는 구조를 먼저 데이터화한다.

이 기준이 유지되면 PSI는 안전관리 도구를 넘어, `건설현장의 사고 전조 해석 엔진`으로 발전할 수 있다.
