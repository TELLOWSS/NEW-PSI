# PSI 본질 재정의 및 상품화 UI 대수술 계획

작성일: 2026-07-09

## 1. 현재 포지션

PSI의 상품 포지션은 **OCR+AI 기술을 융합한 위험성평가교육 맞춤 솔루션**이다.

다만 대외 설명에서는 OCR이 주인공이 되면 안 된다. OCR은 출발점이고, 상품의 본질은 다음 흐름이다.

> 수기로 작성된 위험성평가 기록지 1장을 읽어, 근로자가 무엇을 위험으로 이해했는지 확인하고, 관리자의 보호 판단을 거쳐, 모국어 안내와 개인별 위험성평가교육으로 환류하는 시스템.

즉 PSI는 “문서를 읽는 프로그램”이 아니라 **근로자의 위험 이해 수준을 교육과 추적관리로 바꾸는 현장 안전교육 인텔리전스**다.

## 2. 확인검증 결과

확인한 화면과 코드 기준으로 현재 문제는 기능 부족보다 **노출 순서와 언어의 문제**가 크다.

| 영역 | 확인된 상태 | 상품화 관점 문제 |
|---|---|---|
| PSI 소개 화면 | 3단계 상품화 흐름과 2분 증명모드는 정리됨 | 나머지 주요 운영 화면도 같은 기준으로 통일 필요 |
| OCR 분석 목록 | `상세 판단`, `보호 리포트`, `원문 다시 읽기`, `OCR 신뢰도`, `응답품질`, `전체 백업 JSON` 등이 혼재 | 현장 관리자는 “지금 무엇을 해야 하는지”보다 내부 판독 지표를 먼저 보게 됨 |
| 근로자 상세 판단 | `간단 보기 / 상세 보기 / 프로 검증` 구조는 이미 있음 | 기본 보기 안에도 `원문 인식률`, `무결성 점수`, `증빙 고유값`, `종합역량 점수(P)` 등 개발자성 정보가 노출됨 |
| 리포트/증빙 | JSON, CSV, ZIP, manifest 등 증빙 기능은 강함 | 바이어/현장 사용자에게는 먼저 “근로자에게 어떤 교육을 해야 하는가”가 보여야 함 |
| 대시보드 | 사용자가 현재 기존 구성에 만족 | 이번 대수술 범위에서 제외하고, 다른 화면과 용어가 충돌할 때만 최소 정리 |
| 전체 UX | 기능은 많고 정밀함 | 개발자와 제작자만 전체 맥락을 아는 느낌이 생김 |

## 3. 본질 기준 문장

대외용 한 문장:

> PSI는 수기 위험성평가 기록지를 OCR+AI로 분석하고, 관리자 검증을 거쳐 근로자별 모국어 안전교육과 월별 추적관리로 이어주는 위험성평가교육 맞춤 솔루션입니다.

현장 관리자용 한 문장:

> 오늘 기록지를 넣으면, 누가 무엇을 위험하게 봤는지와 어떤 교육을 바로 해야 하는지가 정리됩니다.

바이어용 한 문장:

> PSI는 작성된 서류를 보관하는 것이 아니라, 근로자의 위험 인식 데이터를 교육·개선·추적자료로 전환합니다.

개발자용 한 문장:

> PSI는 OCR 결과, Q1~Q5 구조화, 관리자 수정 이력, 모국어 안내, 개인 안전역량 지표, 월별 환류 데이터를 같은 근거 체계로 유지하는 폐루프 안전교육 시스템입니다.

## 4. 핵심 설계 원칙

1. 기본 화면은 상품모드로 둔다.
   - 처음 보는 사람은 “기록지 → 판단 → 교육 → 추적”만 보면 된다.

2. 개발자 정보는 숨기되 없애지 않는다.
   - `OCR 신뢰도`, `무결성 점수`, `증빙 고유값`, `JSON`, `W1~W6`, `harness`, `secondPass` 등은 개발자/감사 모드에서만 노출한다.

3. 화면마다 “오늘 할 일”을 먼저 보여준다.
   - 분석 화면은 “확인 필요 / 교육 필요 / 리포트 생성”으로 시작해야 한다.

4. 근로자 상세 판단은 “판독 결과”가 아니라 “보호 판단”이어야 한다.
   - 질문은 “이 사람에게 무엇을 설명해야 하는가?”로 바꾼다.

5. 관리자 수정은 모든 결과에 동기화되어야 한다.
   - 한국어 보호 해석 수정 → 모국어 안내 수정 → 개인 안전역량 재계산 → 리포트/월별 추적 반영까지 한 흐름으로 검증한다.

## 5. 화면 모드 체계

### 상품모드

대상: 바이어, 공모전 심사자, 처음 보는 현장 관리자

보이는 것:
- 기록지 1장
- 근로자가 적은 핵심 위험
- 관리자 확인 필요 여부
- 바로 할 교육/안내
- 리포트/월별 추적 연결

숨길 것:
- OCR 신뢰도 수치
- integrityScore
- evidenceHash
- JSON/manifest/CSV
- W1~W6 내부 코드
- harness/workflow/secondPass 내부 상태

### 운영자모드

대상: 실제 현장 안전관리자

보이는 것:
- 수정 입력
- 보호 판단 확정
- 보완 요청
- 원문 다시 읽기
- 리포트 생성
- 근로자별 교육 안내

단, 문구는 기술어보다 업무어를 사용한다.

### 개발자/감사모드

대상: 개발자, 품질검증자, 내부 감사, 제출 증빙 준비자

보이는 것:
- OCR 신뢰도
- 원본-분석 일치도
- 증빙 고유값
- JSON/CSV/ZIP
- 모델/정책 버전
- 점수 가중치
- 재분석 이력
- 실패 코드

## 6. 우선 수술 대상

### 1순위: 근로자 상세 판단 화면

현재 이름:
- 상세 판단

상품화 이름:
- 근로자 보호 판단

기본 화면 구성:
1. 원본 기록지
2. 이 근로자가 적은 핵심 위험
3. 관리자가 확인할 3가지
4. 오늘 전달할 교육 문장
5. 확정 / 보완 / 리포트 버튼

기본 화면에서 숨길 항목:
- 원문 인식률
- 무결성 점수
- 증빙 고유값
- 종합역량 점수(P)
- 등급 기준 상세식
- 세부 루브릭

이 항목들은 `검증 정보 펼치기` 또는 `개발자 검증` 안으로 이동한다.

### 2순위: OCR 분석 목록

현재 문제:
- 목록에 너무 많은 정보와 버튼이 동시에 보인다.
- `상세 판단 바로보기`, `보호 리포트`, `이 기록 삭제`, `원문 다시 읽기`가 같은 무게로 보인다.

상품화 기본 구조:
1. 오늘 확인할 기록
2. 교육 안내가 필요한 기록
3. 리포트 생성 가능한 기록
4. 다시 촬영/재분석이 필요한 기록

버튼 재정리:
- 기본 버튼: `보호 판단 열기`
- 보조 버튼: `교육/리포트 만들기`
- 문제 있을 때만: `원문 다시 읽기`
- 삭제는 더보기 또는 운영자모드로 이동

### 3순위: 리포트 화면

현재 강점:
- 증빙 패키지 기능이 강함

상품화 기본 구조:
- 개인 리포트 생성
- 모국어 안내 확인
- 월별 교육자료 반영

개발자/제출용:
- JSON/CSV/PDF/ZIP/manifest 검증 패키지는 개발자/감사모드로 분리

### 제외: 대시보드

사용자가 현재 기존 대시보드 구성에 만족하고 있으므로 이번 상품화 UI 대수술 범위에서 제외한다.

다만 다른 화면의 용어가 바뀌면서 대시보드와 표현 충돌이 생기는 경우에만 최소 문구 정리를 한다.

2026-07-09 추가 결정:
- `12채널 허브`는 기능이 중복되어 별도 상품 진입점으로 유지하지 않는다.
- 대시보드 구성 자체는 유지하되, 상단 빠른 버튼/좌측 메뉴/메뉴 구성/12채널 명칭은 제거한다.
- 기존 현장 모바일 흐름은 `현장 모바일 실행 흐름`으로 표현해 OCR 분석, 보호 판단, 교육 리포트, 추적관리의 핵심 흐름만 보이게 한다.

## 7. 용어 교체 기준

| 현재 표현 | 상품화 표현 | 개발자 모드 표현 |
|---|---|---|
| OCR 신뢰도 | 원문 인식 상태 | ocrConfidence |
| 무결성 점수 | 원본-분석 일치도 | integrityScore |
| 응답품질 | 위험 이해 수준 또는 교육 필요 신호 | safetyScore |
| 상세 판단 | 근로자 보호 판단 | RecordDetailModal |
| 보호 리포트 | 개인 안전교육 리포트 | evidence report |
| 원문 다시 읽기 | 원본 재판독 | OCR retry |
| 전체 백업 JSON | 전체 백업 | full JSON backup |
| W1~W6 | 안전역량 가중치 | weight keys |

## 8. 검증 기준

상품모드 자동 검증:
- 기본 화면에서 `OCR 신뢰도`, `integrityScore`, `evidenceHash`, `JSON`, `W1`, `harness`, `secondPass`가 보이지 않아야 한다.
- 기본 화면에는 `기록지`, `보호 판단`, `교육`, `리포트`, `추적`이 보여야 한다.
- 근로자 상세 화면의 기본 제목은 `근로자 보호 판단`이어야 한다.
- `개발자 검증 펼치기`를 누르면 기존 기술 정보가 보여야 한다.

브라우저 검증:
- 소개 화면: 3단계 상품화 흐름이 먼저 보이는지
- OCR 목록: 한 기록당 기본 행동이 1~2개로 줄었는지
- 근로자 상세: 10초 안에 “무엇을 확인하고 어떤 교육을 해야 하는지” 보이는지
- 리포트: 개인 교육 리포트 생성 흐름이 JSON보다 먼저 보이는지

현장 검증:
- 기존 OCR 분석 샘플 10건으로 다시 확인
- 관리자 수정 전/후 모국어 안내와 안전역량 점수 재계산 확인
- 바이어 설명 기준 2분 시연 녹화
- 현장 관리자 1명이 설명 없이 버튼을 찾아 실행 가능한지 확인

## 9. 가장 효율적인 실행 순서

### Phase 1. 상품모드/개발자모드 분리 계약 만들기

작업:
- 전역 모드 기준 정의
- 상품모드에서 숨길 기술어 목록 정의
- 자동 검증 스크립트 추가

완료 기준:
- 빌드와 검증에서 기술어 기본 노출을 막는다.

### Phase 2. 근로자 상세 판단 화면 대수술

작업:
- `상세 판단`을 `근로자 보호 판단`으로 변경
- 기본 화면을 5블록으로 재구성
- 기술 정보는 `개발자 검증` 접힘 영역으로 이동
- 관리자 수정 시 모국어 안내/안전역량 재계산 상태를 한 줄로 보여줌

완료 기준:
- 기본 화면에서 `원문 인식률`, `무결성`, `증빙 고유값`, `P 점수`가 바로 보이지 않는다.
- 관리자는 `확정`, `보완 요청`, `교육 리포트`만 먼저 본다.

### Phase 3. OCR 분석 목록 단순화

작업:
- 목록을 `확인 필요`, `교육 필요`, `리포트 가능`, `재판독 필요`로 재분류
- 버튼을 `보호 판단 열기` 중심으로 정리
- 삭제/JSON/세부 재분석은 운영자 또는 개발자 영역으로 이동

완료 기준:
- 한 행에서 핵심 행동이 1개, 보조 행동이 1개 이하로 보인다.

### Phase 4. 리포트/월별 추적 화면 상품화

작업:
- 개인 안전교육 리포트와 모국어 안내를 기본 흐름으로 전면 배치
- 증빙 패키지 기능은 개발자/제출용 영역으로 분리

완료 기준:
- 사용자는 “근로자에게 무엇을 전달할지”를 먼저 본다.

### Phase 5. 2분 증명 시나리오 고정

작업:
- 기록지 1장 선택
- OCR 분석
- 근로자 보호 판단
- 모국어 안내/리포트
- 월별 추적자료

완료 기준:
- 공모전/바이어 앞에서 2분 안에 실제 화면으로 설명 가능

## 10. 결론

PSI는 이미 기능의 깊이는 충분하다. 지금 필요한 것은 기능 추가보다 **노출 순서의 수술**이다.

가장 좋은 방향은 다음이다.

1. 기본은 상품모드로 단순화한다.
2. 운영자는 해야 할 행동만 먼저 본다.
3. 개발자와 증빙 정보는 접힘/개발자모드로 보낸다.
4. 근로자 상세 판단 화면을 가장 먼저 수술한다.
5. 그 다음 OCR 목록과 리포트 순서로 같은 원칙을 확장한다.
6. 대시보드는 현재 구성 유지 대상이다.

첫 구현은 `근로자 보호 판단 화면`부터 시작하는 것이 가장 효율적이다. 이 화면이 PSI의 심장이고, 여기서 “OCR+AI가 교육 맞춤 솔루션으로 바뀌는 장면”이 가장 직접적으로 증명된다.
