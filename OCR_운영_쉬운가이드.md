# OCR 운영 쉬운 가이드 (공종/팀 기준)

## 1) 핵심 개념 (어려운 용어를 쉬운 말로)
- 기록 양식: OCR 분석 후 저장할 항목 묶음(예: 이름, 공종, 위험요인, 조치사항)
- 공종/팀 그룹: 양식을 적용할 대상 그룹(예: 철근 1팀, 형틀 A조)
- 배정: 어떤 그룹에 어떤 양식을 쓸지 연결하는 설정

즉, **양식 만들기 → 그룹 만들기 → 양식 배정** 순서로 보면 됩니다.

---

## 2) 화면에서 하는 일
OCR 분석 화면의 하단 관리 영역은 아래 두 박스로 구성됩니다.

1. 기록 양식 관리
- 양식 이름, 버전, 입력 항목 목록을 등록/수정/삭제

2. 공종·팀별 기록 양식 배정
- 공종/팀 그룹을 등록
- 그룹 + 양식 + 적용일을 선택해서 배정 저장

---

## 3) 운영 순서 (실무 권장)
1. 먼저 기록 양식 1~2개를 등록
2. 실제 현장 공종/팀 이름으로 그룹 등록
3. 각 그룹에 사용할 양식을 배정
4. OCR 분석 진행 후 결과 확인
5. 필요 시 배정 상태를 ACTIVE/INACTIVE로 전환

---

## 4) DB 적용 안내 (중요)
현재 시스템은 중단 없는 전환을 위해 구 용어(company)와 신 용어(group)를 함께 지원합니다.

- 앱은 기본적으로 `group` 뷰/컬럼을 먼저 조회하고, 없으면 자동으로 기존 `company` 구조로 폴백합니다.

### 먼저 실행
- supabase_master_template_migration.sql

### 그 다음 실행 (이번 업데이트)
- supabase_master_group_compat_migration.sql
- supabase_master_group_postcheck.sql (적용 확인)

두 번째 파일은 다음을 처리합니다.
- assignments 테이블에 `group_id` 병행 컬럼 추가
- `company_id`와 `group_id` 자동 동기화 트리거 추가
- `record_master_groups` 뷰 제공

운영자는 아래 문서를 따라 순서대로 실행하면 됩니다.
- SUPABASE_GROUP_MIGRATION_CHECKLIST.md
- SUPABASE_GROUP_CUTOVER_RUNBOOK.md (최종 전환 당일)
- supabase_master_group_cutover_rollback.sql (문제 발생 시 즉시 복구)

## 5) 자주 묻는 질문
Q. 왜 company라는 이름이 아직 DB에 남아있나요?
- 기존 코드/데이터를 끊김 없이 유지하기 위한 호환층입니다. 화면과 내부 로직은 group 기준으로 전환 중입니다.

Q. 지금 당장 운영에 영향 있나요?
- 없습니다. 호환 방식이라 기존 데이터는 그대로 사용 가능합니다.

Q. 나중에 완전 전환도 가능한가요?
- 가능합니다. 충분히 안정화된 뒤 최종 rename 마이그레이션을 추가하면 됩니다.
